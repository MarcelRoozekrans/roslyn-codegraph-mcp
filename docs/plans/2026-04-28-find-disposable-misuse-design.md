# `find_disposable_misuse` ŌĆö Design

**Date:** 2026-04-28
**Backlog source:** `docs/BACKLOG.md` ┬¦ 3 (async & concurrency tools)
**Sibling tool:** `find_async_violations` (same shape, same audience)

## Goal

Detect `IDisposable` and `IAsyncDisposable` instances that risk leaking ŌĆö local variables holding a disposable that aren't wrapped in `using`/`await using`/returned/assigned to a field, plus bare-expression-statement discards. Static analysis only; no runtime tracking, no fix suggestions.

## Why

Resource leaks (file handles, DB connections, sockets, `HttpClient` channels) are a common production bug source in .NET. Modern conventions (`using` declarations, `await using`) make leaks easy to avoid, but easy to forget. The .NET analyzers that catch some of these (CA2000) aren't on by default in many projects. This tool surfaces them as a single MCP call producing a triage-friendly list.

## Two patterns

### 1. `DisposableNotDisposed` ŌĆö severity `warning`

**Pattern:** A `LocalDeclarationStatementSyntax` initializing a disposable type, where:
- The declaration is **not** a `using` declaration
- The variable is **not** later returned by the method
- The variable is **not** assigned to a field, property, or out-parameter

**Detection algorithm:** for each method body, walk every `LocalDeclarationStatementSyntax`. For each declarator with an initializer:
1. If the local declaration's `UsingKeyword` is set ŌåÆ handled (skip).
2. Look up the declarator's variable type (or initializer's type). Is it `IDisposable` or `IAsyncDisposable` (via `AllInterfaces`)? If not ŌåÆ skip.
3. Walk the rest of the method body for any of:
   - `ReturnStatementSyntax` whose expression is the variable identifier (or wraps it in casts/expressions referencing it)
   - `AssignmentExpressionSyntax` LHS is a `MemberAccessExpressionSyntax` (`this.field`, `instance.Property`) and RHS is the variable
   - `AssignmentExpressionSyntax` LHS is an out-parameter and RHS is the variable
4. If none of the disposal acknowledgments are found ŌåÆ flag.

**Severity is `warning`** ŌĆö sometimes the developer has a long-lived ownership pattern not visible to syntactic analysis. Real teams accept the occasional false positive in exchange for catching the common bugs.

### 2. `DisposableDiscarded` ŌĆö severity `error`

**Pattern:** An `ExpressionStatementSyntax` whose expression returns a disposable type and whose result is implicitly discarded (no assignment, no `_ = `, no using, no return).

```csharp
new FileStream(path, FileMode.Open);  // FLAG ŌĆö created and immediately leaked
File.OpenRead(path);                   // FLAG ŌĆö same shape, factory variant
```

**Detection algorithm:** walk every `ExpressionStatementSyntax`. If `Expression` is an `ObjectCreationExpressionSyntax` or `InvocationExpressionSyntax` whose return type implements `IDisposable` / `IAsyncDisposable`, flag.

**Severity is `error`** ŌĆö this is always a bug. There's no scenario where creating a disposable and immediately discarding it is intentional.

## Disposability check

A type is "disposable" if its `INamedTypeSymbol` (or any base type / interface) is `System.IDisposable` or `System.IAsyncDisposable`.

Implementation: precompute `idisposableSymbol` and `iasyncDisposableSymbol` per compilation. For each candidate type, walk `AllInterfaces` looking for either; or check `BaseType` chain for the interface itself.

---

## Algorithm shape

Single pass per non-test compilation. For each method declaration:
1. Walk `LocalDeclarationStatementSyntax` nodes for pattern 1.
2. Walk `ExpressionStatementSyntax` nodes for pattern 2.
3. For each pattern-1 candidate, scan the method body for disposal-acknowledgment syntax (return / field assign / out param).

Performance: O(method bodies ├Ś method body size). Comparable to `find_async_violations`.

## Edge cases

| Case | Behaviour |
|------|-----------|
| `using var x = ...;` | Not flagged (handled). |
| `using (var x = ...) { ... }` | Not flagged (handled). |
| `await using var x = ...;` | Not flagged (handled). |
| `var x = ...; return x;` | Not flagged (caller takes ownership). |
| `var x = ...; this._field = x;` | Not flagged (instance owns it). |
| `var x = ...; outParam = x;` | Not flagged (out parameter, caller owns). |
| `var x = ...; try { ... } finally { x.Dispose(); }` | **Flagged** in v1 ŌĆö legacy pattern, modern code uses `using`. Documented limitation. |
| `var reader = new StreamReader(new FileStream(...))` (no `using`) | `reader` is flagged as `DisposableNotDisposed`. Inner `new FileStream(...)` is not separately flagged (its surrounding declaration covers it). |
| `using var reader = new StreamReader(new FileStream(...))` | Not flagged ŌĆö outer `using` covers both per StreamReader's documented ownership. |
| `_ = new FileStream(...);` | Not flagged ŌĆö explicit discard suggests the developer is aware (rare anti-pattern, but explicit). |
| `var x = new T(); var y = x;` (aliasing) | `y` is checked separately; we don't track aliasing. If `y` is then disposed/returned, `x` is still flagged. Documented limitation. |
| `IDisposable` interface variable | Treated identically to concrete types ŌĆö checks if type implements the interface. |
| Type that implements both `IDisposable` and `IAsyncDisposable` | Either `using` or `await using` counts as handled. |
| Method argument: `DoSomething(new FileStream(...))` | Not flagged in v1 (out of scope). Documented as future-work. |
| `IDisposable` field declaration not disposed in `Dispose()` | Not flagged ŌĆö that's a separate concern (CA1001) outside this tool's scope. |

## Tests

### Fixture additions

Add to existing `AsyncFixture` project (already a production project, no test framework refs) or create a new `DisposableFixture` project. **Recommend new project** to keep concerns clean ŌĆö `DisposableFixture/Misuse.cs` containing positive and negative cases.

`DisposableFixture/DisposableFixture.csproj` ŌĆö minimal SDK-style, `IsPackable=false`, no test framework refs.

`DisposableFixture/Misuse.cs` ŌĆö pseudocode:

```csharp
using System;
using System.IO;
using System.Threading.Tasks;

namespace DisposableFixture;

public class Misuse
{
    // ============ POSITIVE CASES ============

    // Pattern 1: local declaration not disposed (constructor)
    public void NotDisposedFromConstructor()
    {
        var stream = new MemoryStream();
        stream.WriteByte(1);
    }

    // Pattern 1: local declaration not disposed (factory)
    public void NotDisposedFromFactory()
    {
        var stream = OpenStream();
        stream.WriteByte(1);
    }

    // Pattern 2: discarded constructor
    public void DiscardedConstructor()
    {
        new MemoryStream();
    }

    // Pattern 2: discarded factory
    public void DiscardedFactory()
    {
        OpenStream();
    }

    // ============ NEGATIVE CASES (must NOT be flagged) ============

    public void DisposedViaUsingDeclaration()
    {
        using var stream = new MemoryStream();
        stream.WriteByte(1);
    }

    public void DisposedViaUsingStatement()
    {
        using (var stream = new MemoryStream())
        {
            stream.WriteByte(1);
        }
    }

    public async Task DisposedViaAwaitUsing()
    {
        await using var stream = new MemoryStream();
        await stream.WriteAsync(new byte[] { 1 });
    }

    public Stream ReturnedToCaller()
    {
        var stream = new MemoryStream();
        return stream;
    }

    private MemoryStream? _heldField;
    public void StoredInField()
    {
        var stream = new MemoryStream();
        _heldField = stream;
    }

    public void StoredInOutParam(out Stream result)
    {
        var stream = new MemoryStream();
        result = stream;
    }

    public void DiscardWithUnderscore()
    {
        _ = new MemoryStream();  // explicit acknowledgment, not flagged
    }

    private static MemoryStream OpenStream() => new MemoryStream();
}
```

## Out of scope (explicit non-goals)

- **Method-argument disposables** (`DoSomething(new FileStream(...))`) ŌĆö too ambiguous in v1.
- **try/finally + explicit `Dispose()`** ŌĆö legacy pattern, will be flagged as false positive on older codebases. Defer.
- **Wrapper-pattern ownership inference** (`StreamReader(stream)` owning inner) ŌĆö only the outer variable's lifetime is tracked.
- **`IDisposable` field not disposed in `Dispose()`** ŌĆö that's CA1001 territory; out of scope.
- **Aliasing tracking** ŌĆö `var y = x;` doesn't propagate disposal acknowledgment back to `x`.
- **Conditional disposal flow analysis** ŌĆö if a variable is disposed in one branch but not another, we don't analyze. Either fully disposed or flagged.
- **Lambdas / local functions / async methods** ŌĆö same coverage as `find_async_violations`: walk the outer method's body, attribute violations to the outer method. Nested-function-scope analysis is a follow-up.
- **No suppression mechanism** ŌĆö agents post-filter the JSON.

## Decisions log

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Patterns | `DisposableNotDisposed` (warning) + `DisposableDiscarded` (error) | Mirror find_async_violations' "always-bug = error, usually-bug = warning" pattern |
| Output shape | Summary + flat violations list | Established convention |
| Scope | Whole-solution, production-only | Same as find_async_violations |
| Method-argument disposables | Skip in v1 | Too ambiguous (caller may dispose) |
| try/finally with explicit Dispose | Flag (legacy false positive accepted) | Modern code uses `using`; document limitation |
| Wrapper-pattern recognition | Not modeled | Outer-variable tracking catches the leak; inner constructor isn't separately flagged |
| Aliasing | Not tracked | Each local variable analyzed independently |
| Reuse `IsGeneratedFile` | Extract to shared helper if convenient | DRY with find_async_violations |
| Severity counts | Two: warning + error | Same enum / same convention |
